# Flying Run 项目分析文档

本分析文档基于 Flying Run 项目的现有代码架构以及业务逻辑，对底层支撑、技术选型以及四大核心业务模式提出具体的落地细节与架构规划。

---

## 1. 整体架构定位
**本 App 仅仅作为一个“富前端 (Rich Frontend)”交互界面层。** 
所有的核心计算、数据流转均交由服务端处理。App 端内部通过 `ApiConfig`（位于 `lib/config/api_config.dart`）作为统一的网关路由，集中调用各服务厂家以及自研后端的 API 进行数据分析与交互。

---

## 2. 核心大模型与算法服务选型 (成本预估与方案)

### 2.1 医学健康数据分析模型
*   **技术组合**：`QLoRA` + `Qwen3-7B` + `DPO 对齐`
*   **实施逻辑**：
    1.  采集用户端产生的生理数据（心率、步数、睡眠时长）以及第三方硬件数据。
    2.  利用开源的千问模型 (`Qwen3-7B`) 作为底座，采用 `QLoRA` 降低显存占用与微调成本。
    3.  最后通过 `DPO (Direct Preference Optimization)` 对齐医学与健康管理专家的偏好，确保给出的“运动处方”和饮食建议科学且安全。
*   **成本预估**：硬件租赁与微调训练成本控制在 **5 万元人民币以内**。

### 2.2 AI 陪伴式教练语音交互
*   **技术方案**：云端 `GPT-SoVITS`
*   **实施逻辑**：
    1.  放弃端侧低质量的 TTS 模拟，全量采用基于云端 GPU 集群的 `GPT-SoVITS` 极速声音克隆方案。
    2.  只需少量用户（或教练）的高质量录音，即可实现高精度的音色与情感复刻。
    3.  **安全性与隔离性**：账户与音色严格绑定配对（Token 验证），保证自定义音色的安全性与专有性。
*   **成本预估**：API 部署与推理成本控制在 **1 万元人民币以内**。

## 3. 核心业务模式架构拆解

### 3.1 校园模式 (智慧体育与合规性)
*   **鉴权与数据合规 (核心安全逻辑)**：
    1.  采用联合鉴权机制，通过调用校园方提供的 API 进行身份认证。
    2.  **数据不落地原则**：对于学生产生的校园跑数据，我们的 App 和服务器仅仅作为**数据传输中转站 (Relay)**。数据流通过加密通道直接流向校园内部的私有服务器，不进入并留存在我们的中心数据库。以此彻底规避学生敏感数据的合规与隐私风险。
*   **功能实现路径**：
    1.  **GPS 定位与打卡机制**：在 App 端开发定位模块（如通过 `geolocator`），记录运动轨迹以实现防作弊。
    2.  **动态任务下发**：对接学校发布的任务 API。校方教务人员配置每日跑步任务（包括规定区域、配速、公里数），下发至我们的服务器，再由我们的服务器统一下发给对应校园的用户终端。
    3.  **智能硬件联动**：对于不携带手机跑步的学生，支持调用对应智能手表/手环厂商的公开 API，读取步频、配速和心率数据，完成校园内跑任务。

### 3.2 社区模式 (体医融合与适老化)
*   **业务逻辑**：专注老年人群慢病管理与实时健康保护。
*   **功能实现路径**：
    1.  依托公司自研的“客服/全科医生接口”，搭建老年人保护直连通道。
    2.  **穿戴设备强依赖**：用户需佩戴兼容的智能手表，App 实时拉取并监测老人的心率、血氧以及跌倒状态等体征信息。
    3.  一旦监测到心率异常或突发状况，App 自动通过 API 报警并一键接通社区卫生院客服或绑定的监护人。

---

## 4. 统一消息推送系统 (Push Notifications)
为了保障三大模式（标准、校园、社区）的正常运转，必须构建全模式统一的推送触达机制：
1.  **校园模式推送**：定向提醒每日校园阳光跑任务下发、打卡截止预警等。
2.  **普通模式推送**：全局推送 AI 伴跑周报生成通知、久坐提醒等。
3.  **社区模式推送**：紧急健康预警、社区康养活动通告等。
4.  *(实施建议：可接入极光推送、各大手机厂商级通道保证后台保活到达率。)*
